## Successes & Strengths

1. **100% story completion** — all 4 stories delivered with clean CI (lint, typecheck, test, build, schema drift)
2. **Front-loaded tooling pain** — 13 debug issues solved permanently across 4 stories. Svelte 5 + Vite gotchas (ESM compatibility, `.svelte.ts` ESLint parsing, `$lib` alias, Vitest browser resolution, `$state` proxy behavior, `$effect` module scope) are all resolved for future stories
3. **Strong test infrastructure** — co-located tests, test count growing every story (14 → 17 → 50 → 68), Vitest properly configured for Svelte 5 components
4. **Key patterns established:**
   - Lazy-init Supabase client (avoids `loadConfig()` timing issue)
   - `(select auth.uid())` RLS optimization
   - Typed API client with `ApiResult<T>` discriminated union and 401 retry
   - Auth middleware with 60s JWT verification cache
   - `resolve.conditions: ['browser']` for Svelte 5 Vitest
5. **Clean CI pipeline** — lint, typecheck, unit test, build, and schema drift gates all operational

## Challenges & Growth Areas

1. **High review finding volume** — 25 findings across 4 stories (~6/story). Story 1.1 had 12 alone. Categories:
   - Packaging/build config (5): Docker contexts, shared package exports, Turbo pipeline deps, .gitignore
   - Accessibility (4): Landmark exposure, skip link focus, focus-visible treatment
   - Test quality (3): Instance isolation, module state reset, DOM environment
   - Schema/API correctness (4): NULL alignment, validation, auth middleware scope
   - Security (2): JWT cache keying, error message exposure
   - Design tokens (2): Typography gaps, import-order rule
   - Logic bugs (2): 401 retry null session, CI drift stderr
2. **Two-step local dev bootstrap** — `supabase start` and `pnpm dev` are separate commands; no single-command setup
3. **Svelte 5 tooling friction** — 7 of 13 debug issues were Svelte 5 or Vite-specific; the "no SvelteKit" decision was conscious and correct but came with upfront cost
4. **Scattered knowledge** — debug solutions live in individual story dev notes, not consolidated

## Key Insights & Learnings

1. **Front-loaded tooling pain pays off** — solving tooling issues in Epic 1 means Epic 2 stories inherit a stable foundation
2. **Build/packaging is the blind spot** — application logic was generally correct on first pass; export configs, Docker contexts, and CI steps needed the most corrections
3. **Accessibility needs upfront attention** — consistently under-delivered on first pass; will compound in Epic 2 where every story ships UI components
4. **Test quality improved over the epic** — Story 1.1 tests needed the most review fixes; by Story 1.4 patterns were cleaner
